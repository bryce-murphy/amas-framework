# AMAS v3 Design Brief

## Executive recommendation

AMAS v3 should be a **split package**, not a simple consolidation release and not a single monolith. The strongest shape is: **a smaller canonical core framework, a GitHub reference implementation, a usage guide, and versioned templates plus GitHub Actions**. That recommendation follows directly from the live UPCDS state: the canonical AMAS document on `main` has moved to v2.14.1, but the surrounding instruction and workflow surfaces show drift, and the most recent merged PRs are dominated by framework adoption, PMN evidence capture, and prompt-reference/tooling governance rather than domain implementation. In other words, v2.x is carrying too much cross-cutting load in too few canonical places, and each upgrade is creating more ŌĆ£surface synchronizationŌĆØ work than the current architecture cleanly supports. ## External research findings

The external tooling research strongly supports making **Context7 and MCP first-class, but conditional, workflow components**. Context7ŌĆÖs official docs describe it as a way to pull up-to-date, version-specific library documentation and code examples directly into an AI coding assistant; its own best-practices pages recommend making it a rule for library/API documentation, code generation, setup, and configuration work, and its recommended install path is a remote MCP server with API key support. Context7 also states that full prompts and source code stay with the assistant, while the API only receives assistant-formulated documentation queries, library identifiers, and related metadata. That makes Context7 a very good fit for dependency lookup, framework configuration, and migration work, but a poor candidate for ŌĆ£always-on for every taskŌĆØ usage. In AMAS v3 terms, Context7 should be **recommended by default for fast-moving external APIs and version-specific framework work, optional for general coding, and unnecessary for purely repo-internal reasoning already grounded in GitHub artifacts**. The MCP research points in the same direction. The official MCP documentation says authorization is optional overall, but when HTTP transport is used, implementations should follow the authorization specification; stdio transport should instead get credentials from the environment. ## Proposed AMAS v3 architecture

The recommended architecture is a **four-layer package**. **Layer one, Core AMAS v3**, is canonical law. It should define: GitHub-as-canonical-memory; artifact classes; role model; capability-class scorecard; kickoff and re-score rules; universal handoff schema; review/adjudication model; research/write-back model; deterministic-enforcement principle; and framework-adoption/surface-sync principle. **Layer two, GitHub reference implementation**, should define concrete branch/PR/CODEOWNERS/review/workflow behavior. **Layer three, usage guide**, should tell people how to run greenfield, retrofit, solo, and small-team projects without over-bloating process. **Layer four, versioned templates and actions**, should ship actual starter files and reusable workflows. That is the cleanest way to keep v3 GitHub-specific while still preventing the canonical framework from becoming a running scrapbook of product quirks. ŅłĆfileciteŅłéturn4file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn6file0ŅłéL1-L1Ņłü

The **core role model** should keep **Architect, Builder, and Reviewer as required roles**, because those roles are still the smallest set that preserves planning, execution, and independent checking. AMAS v3 should then add **Researcher, Release Manager, Tooling/Automation Agent, and Adjudicator** as optional roles, not new mandatory ones. The internal UPCDS scorecard already demonstrates the practical value of explicit scoring and re-score triggers, but it is still too vendor/surface-specific for v3 core. V3 should instead score against capability dimensions such as reasoning quality, repo-read, repo-write, terminal execution, PR authoring, review quality, CI/log awareness, MCP access, Context7/doc retrieval, web research, cost, latency, and domain fit. Re-score triggers should include major model/tool changes, MCP access changes, review-quality regressions, project-type changes, and a standing 90-day cadence. ## Proposed review robustness and deterministic enforcement

AMAS v3 should define a **two-part review robustness model**. The **core law** should require that every substantive AI review be attributable to a specific bot/surface, a specific head commit, and a specific output class, and that claimed actions be separable from substantive findings. The **GitHub reference implementation** should then supply the polling, API checks, and workflow templates that operationalize that rule. That split matches both the UPCDS evidence and the current platform ecosystem. PR #24 and PR #26 show why the separation is needed: a bot can produce real substantive analysis at the same time that its claimed commit/file/PR side effects are phantom. ŅłĆfileciteŅłéturn20file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn30file0ŅłéL1-L1Ņłü

For `@codex review`, my exact recommendation is this. **Invoke explicitly, capture the invocation artifact, poll both PR reviews and issue comments, and validate against the current head SHA before merge.** When output arrives, classify it as one of four states: formal PR review with findings, formal PR review without findings, issue comment sentinel/advisory, or claimed-action output. If the bot claims to have created or changed something, run four checks: claimed commit existence, claimed file existence, claimed follow-up artifact existence, and claimed identifier-pattern correctness. If a clean signal is older than the current head, it is stale. If a bot only comments on the PR conversation without a review object, that may still be substantive evidence, but it should not be mistaken for an approval. If new code-modifying commits land, require re-review or a bypass comment that explicitly names why freshness is being waived. That is the most direct generalization of the repoŌĆÖs own working practice. ŅłĆfileciteŅłéturn20file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn22file0ŅłéL1-L1Ņłü ŅłĆfileciteŅłéturn30file0ŅłéL1-L1Ņłü

| v2.x mechanism | v3 treatment | Why |
|---|---|---|
| GitHub is canonical source of truth | **Keep canonical** | It is the strongest stable rule in the system and aligns with live UPCDS practice. |
| Architect / Builder / Reviewer | **Keep canonical** | Still the minimal role triangle that protects quality. |
